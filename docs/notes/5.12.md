## 明日计划

📅 5.13 执行计划：从数据分层到强化学习闭环
1. 数据集重构：建立“9类题型”平衡矩阵
你已经意识到按答案类型划分数据集存在局限。由于你手头的 9500 条数据呈现长尾分布（3200:3200:...:16），直接训练会导致模型对低频题型丧失推理能力。

题型对齐与采样：

将 9500 条数据按 9 类题型重新归类。

头部大类（3200条类）：每类各抽取 500-800 条 高质量题目即可，避免逻辑过拟合。

尾部稀缺类（16条/400条类）：进行 全量保留，并标记为后续需要 GPT-5.5 精标的“金牌数据”。

构建 Reasoning SFT 种子集：

利用你提到的 RLAIF 思路，调用 DeepSeek-V4 Preview 为筛选后的题型生成 thought（思维链）。

自动化验证：编写脚本，仅保留“生成答案 == 原始答案”的条目，构建包含 Question -> Thought -> \boxed{Answer} 的训练集。

2. 奖励函数（Reward Function）工程化
既然你决定采用 GRPO 策略，奖励函数是代替 Critic 网络的“裁判”。

实现 RewardFunction 类：

准确性奖励（核心）：使用正则提取 \boxed{} 内容并比对原始标签。

格式奖励（防截断）：若输出包含完整的 \boxed{} 结构，给予额外加分（解决 5.11 提到的截断问题）。

逻辑长度奖励：若 thought 部分字数符合预期（非空且逻辑连贯），给予正向反馈，引导模型进行推理侧缩放（Inference-time Scaling）。

3. Reasoning LoRA 注入与配置
针对推理任务，你需要比 chat_lora 更深层的参数更新。

扩展 Target Modules：

在 LoRAConfig 中，除了 attention 层，必须包含 FFN 层：['gate_proj', 'up_proj', 'down_proj']。

原理：数学逻辑的变换规则大量存储在 FFN 层，仅动 Attention 层难以激活深度推理能力。

Rank (r) 调整：尝试将 r 提升至 16 或 32，给复杂的 CoT 逻辑留出足够的表达空间。

🛠️ 关键技术检查清单（Sanity Check）
[ ] Token 对齐复检：确认 \boxed{} 中的 \ 符号不会因为 Tokenizer 处理导致偏移或丢失。

[ ] GRPO 接口准备：为数据增加 group_size 维度，确保一个 Question 对应多个候选 Answer 采样，用于计算相对奖励。

[ ] Reference Model 锁定：将你目前输出尚可的 chat_lora 设为 Reference Model，防止 RL 阶段模型逻辑跑偏。
