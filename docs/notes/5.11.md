## 📅 5.12 详细执行计划：从格式对齐到推理增强

### 1. `chat_lora` 深度打磨：解决“存疑”与“截断”

你目前对编码对应关系存疑，且出现了格式截断（缺少 `\box{}` 或结尾符），这通常是 Tokenization 细节问题。

* **可视化验证（彻底消除存疑）：**
* 编写一个简单的 `debug_tokens.py` 脚本，将一条 `input_ids` 和 `labels` 重新 `decode` 回文字。
* **检查重点：** 确认 `-100` 是否精准覆盖了所有的 System Prompt 和 User Question，且第一个计算 Loss 的 Token 是否恰好是 Answer 的起始位置。


* **修复截断与格式问题：**
* **EOS Token 检查：** 检查你的数据集是否在每个 Answer 结尾添加了 `<|endoftext|>` 或模型特定的 `end_token`。如果 labels 结尾没有 EOS，模型永远学不会自发停止。
* **Padding 策略：** 确认 `max_length` 设置是否足够容纳完整的 `\box{}` 结构。如果是在推理（Inference）时截断，调整 `max_new_tokens`。


* **Loss 收敛监控：**
* 在过拟合 5-20 条数据时，Loss 应该能降到 $10^{-2}$ 甚至更低。如果 Loss 停留在 0.5 左右无法下降，说明 `-100` 的 mask 可能覆盖到了 Answer 部分。



### 2. 构建 `reasoning_lora`：转向逻辑蒸馏

`reasoning_lora` 的目标不是让模型“说话”，而是让模型“思考”。针对你参加的 **NVIDIA Nemotron Challenge**，你需要从 SFT 转向思维链（CoT）的强化。

* **数据构造：引入“思维轨迹”（Thought Traces）：**
* 不要只给模型 `Question -> Answer`，要寻找或构造带有 `Question -> Thought -> Answer` 结构的数据。
* **关键点：** 在 SFT 时，`Thought` 部分也应该计算 Loss，因为它承载了逻辑权重。


* **学习重点：理解推理模型的 scaling law：**
* 了解 **Inference-time Scaling**（推理侧缩放）：即使不微调，增加推理时的采样（Self-consistency）能否提升表现？
* **模型选型建议：** 如果 Nemotron 表现持续不佳，考虑对比同参数量的 Qwen2-Math 或 DeepSeek-Math 的逻辑表现，判断是否需要更换 Base Model。



### 3. 补课：构建“评测脚本”雏形（填补 5.11 欠账）

没有评测，你的训练就是盲目的。

* **实现 Matcher 类：**
* 编写正则提取器，专门提取模型输出中 `\box{}` 里的内容。
* **数学对齐：** 实现一个简单的逻辑，处理 `0.5` 和 `1/2` 是否被视为同一个正确答案（数值等值判断）。


* **批量推理：**
* 写一个循环，对测试集（如 50 条题）进行批量推理并保存为 JSONL，方便肉眼对比“模型答案”与“标准答案”的逻辑差异。
