# 明日计划（重点：跑通 + 可诊断）

## 1️⃣ 先做一件最重要的事：SFT loss 正确性验证（必做）


### 检查点：

* [ ] prompt 和 answer 是否正确拼接
* [ ] label mask 是否正确（重点）

  * prompt → -100
  * answer → 保留 token id
* [ ] loss 是否只在 answer 上下降

### 验证方法（非常关键）：

* 打印一条 batch：

  * input_ids
  * labels
* 确认：

  * prompt 部分 = -100
  * answer 部分 = 有 token

👉 如果这里错了，你后面 RL / LoRA 全是无效实验

---

## 2️⃣ LoRA 结构 sanity check（你现在很可能“装了但没学”）

你已经搭了 chat_lora，但明天要验证：

### 检查点：

* [ ] `model.print_trainable_parameters()`

  * trainable params 是否 < 1%
* [ ] target_modules 是否真的命中

  * q_proj / v_proj / k_proj / o_proj
  * gate_proj / up_proj / down_proj
* [ ] forward 是否发生变化（LoRA 是否注入成功）

👉 如果 trainable params = 0 或极低异常：
说明 LoRA 根本没生效

---

## 3️⃣ 做一个“最小过拟合实验”（核心诊断方法）

这是判断你 pipeline 是否正确的 gold test：

### 做法：

* 只取 5～20 条数据
* 训练 200~500 steps
* 看 loss 是否下降到接近 0

### 结果判断：

* ❌ loss 不降 → pipeline 有 bug（mask / lr / LoRA）
* ⚠️ loss 降但输出不对 → 数据问题
* ✅ loss 收敛 → pipeline 正常

---

## 4️⃣ 分析 Nemotron 输出差的真正原因（不要继续只调 prompt）

你现在已经验证：

> prompt engineering 无效甚至变差

这其实是一个**很重要的信号**：

### 可能原因（按优先级）：

1. ❗ SFT 数据质量问题（最常见）

   * 推理步骤不一致
   * 答案格式不统一
2. ❗ mask 错误（模型在学 prompt）
3. ❗ base model 本身 reasoning 能力弱
4. ❗ decoding 设置问题（temperature / top-p）

👉 明天目标不是修 prompt，而是定位是哪一类问题

---

## 5️⃣ 加一个“评测脚本雏形”（很关键）

你已经在做 scoring，这一步要工程化：

### 做一个最小 evaluator：

* 输入 question
* 输出：

  * model answer
  * gold answer
  * exact match / rule-based score

建议先不用 fancy metrics，就三种：

* exact match
* numeric match
* boxed extract match
